資料庫介紹

表格與資料庫

  • 表格 ( Table )：以欄 ( columns ) 與列 ( rows ) ⽅式來儲存資料
  • 資料庫 ( Database , DB )：儲存著⼤量表格以及相關資料庫物件 ( schemas, tables,views…) 的綜合儲存體
  • 資料庫綱要 ( Database Schema )：定義資料庫組成⽅式的⼀種資料，可以說是資料庫的藍圖

資料庫管理系統 DBMS

  • Database Management System，DBMS
  • 管理資料庫的⼤型軟體
  • 允許資料庫管理師 ( DBA ) 透過命令列或是視窗化介⾯來完成資料庫相關事務
    • 例如：資料存取與權限管理
  • 使⽤者：
    • 資料庫設計師 ( Database Designer )
    • 資料庫管理師 ( Database Administrator, DBA )
    • 應⽤程式設計師 ( Application Designer )
    • ⼀般使⽤者 ( End User )

交易管理 Transaction Management

  • 對資料庫執⾏⼀個交易可能包含⼀連串的新增、修改或刪除指令
  • 為了保證交易的正確與可靠，必須符合 ACID ( Atomicity, Consistency, Isolation, Durability )
    • Atomicity ( 單元性 )：⼀個交易中所含有的所有運算不是完全做完就是完全不做 → 要嘛成功要嘛失敗，不會有做⼀半的情況
    • Consistency ( ⼀致性 )：交易前與交易後的結果都沒有破壞資料庫的完整性，也就是完全符合資料庫設定的規則
    • Isolation ( 隔離性 )：同⼀筆資料在⼀個交易尚未完成前不可讓其他交易執⾏更改 → 預防同時性 ( synchronization ) 可能造成的錯誤
    • Durability ( 永久性 )：⼀旦交易成功，對資料的變更即永久有效，即使系統故障也不影響 → 須仰賴資料庫備份與交易⽇誌 ( log ) → 備份也要考慮距離的遠近

關聯式資料庫 Relational Database

  • 以表格 ( table ) ⽅式來儲存與呈現資料，再以數學集合論為基礎將表格與表格之間建⽴關聯以處理複雜的資料關係
  • SQL 指令就是⽤於關聯式資料庫
  • 有關聯式資料庫，當然也有⾮關聯式資料庫。兩者差異：
    • 關聯式資料庫：節省空間、浪費時間
    • ⼤數據資料庫、⾮關聯式資料庫：節省時間、浪費空間

關聯式資料庫管理系統 RDBMS

  • Relational Database Management System , RDBMS
  • 專⾨⽤來管理關聯式資料庫的系統，現今資料庫管理系統⼤多屬之

為什麼要採關聯式設計？

  • 將所有資料儲存在⼀個表格內會有資料重複問題，如果資料量⼤會造成⾄少以下問題
  1. 浪費儲存空間
  2. 資料錯誤率提⾼
  3. 增加修改上困難

有優點當然也會有缺點。缺點：查詢時較花費時間

主鍵 PK

  • Primary Key，PK
  • 有 PK constraint：主鍵限制 ( 主鍵防呆 )
    • 值不可以重複
    • 值不可以為空值
  • 關聯式資料庫每⼀表格都「應該設定⼀個 PK 」
  • ⼀個表格最多只會有⼀個 PK
  • PK 可以是 ⼀個欄位，也可以是多個欄位組成的複合欄位
  • 識別的欄位，⼈類刻意為之 ( 另外加⼊的 )，每個資料都是獨⼀無⼆的不會重複 → 值不會重複，且不為空值

主鍵建⽴過程

  • 建⽴過程：先找 Super Key → 再找 Candidate Key → 從 Candidate Key 中挑選 PK
  • 超鍵 ( Super Key )：符合唯⼀性
  • 候選鍵 ( Candidate Key )：
    • 符合唯⼀性與最⼩性 ( 欄位越少越好，越多越複雜 )
    • 主鍵是由候選鍵中挑選出來
  • 次要鍵 ( Alternate Key )：沒被選為主鍵的其他候選鍵

外來鍵 Foreign Key，FK

  • 外來鍵欄位是為了關聯⽽存在的
  • 外來鍵欄位必須參照到另⼀個表格的主鍵欄位 ( 或具有唯⼀性的欄位 )，以避免多頭⾺⾞ → 避免不知道要參照到哪個欄位
  • 主鍵欄位必須有值，外來鍵⽅可關聯，以避免斷頭資料

被參照到的資料，就不可以被刪除

建⽴表格 - 設計流程

  • 制定欄位：
    • 表格是主要儲存資料的地⽅，依照資訊系統功能來分析需要儲存哪些資料，並將其轉化成對應的欄位
    • 蒐集資料、蒐集欄位資訊：這樣才知道要建⽴什麼欄位
  • 正規化

資料庫正規化

  • 資料庫正規化 ( Database Normalization )
    • 關聯式資料庫才適⽤
    • 建⽴資料庫內的表格與欄位時所應採⽤的設計法則
    • 減少表格內的資料重複
    • 增進資料的 ⼀致性
  • 正規化種類
    • 第 1 正規化 ( First Normal Form， 1NF )：
      • 每個欄位內都是儲存單⼀值
    • 第 2 正規化 ( Second Normal Form， 2NF )：
      • 符合第 1 正規化
      • ⾮鍵欄位需要與鍵欄位 ( 主鍵與候選鍵 ) 有⾼度的相依性 ( 有沒有關係 )，否則應予移除
    • 第 3 正規化 ( Third Normal Form，3NF )
      • 符合第 2 正規化
      • ⾮鍵欄位間不應有相依性，否則應予移除
      • 1970 年 Edgar Frank Codd 已經提出前 3 個正規化的概念
    • 還有 BC 正規化、第 4 正規化、第 5 正規化、DK 正規化、第 6 正規化 …
    • 滿⾜後⾯的正規化「必定」滿⾜前⾯的正規化，例如滿⾜ 3NF 必定滿⾜ 2NF 與 1 NF
  • 結論
    • ⼀般建議滿⾜ 3NF 即可
    • 滿⾜越多正規化，理論上⽽⾔資料的關聯會有更好的約束性；但也導致表格過多、資料更破碎⽽需要常常合併表格造成執⾏效能差的結果

執⾏效能差的結果

  • 除⾮有很好的原因要違反正規化，不然為了避免資料重複性⽽導致⽇後資料過多⽽降低執⾏效能，要盡量遵守正規化
  • 除⾮要計算的資料「⾮常花時間」，才會建⽴欄位加以儲存，以避免每次要⽤到該資料時，都得再次浪費時間去計算 → 節省未來計算 / 查詢的時間
